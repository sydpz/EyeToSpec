# 三层架构边界宪法(Config / 转换库 / 业务层)

这份文档钉死 EyeToSpec 输出的布局信息,如何一路流到游戏画面上——**每一层做什么、明确不做什么、以后缺能力往哪补**。它是后续开发转换库(②)和业务消费(③)的依据。

底层是一条贯穿所有决策的**总原则**:

> **Config 只存"画出来的视觉事实"(绝对 px 实例 + 对齐意图标注)。一切"关系"和"规律"(重复区间距、弹窗跟随、内容驱动尺寸)由业务层从实例里反推或自己写。转换库只做 config→Phaser struct 的转换 + 一批按需生长的通用函数。**

EyeToSpec 的目标是**对齐视觉稿,不是对齐动效**。它传达的是一张界面的**布局信息**(长什么样、在哪、多大、谁压谁);把布局信息转化成真实代码逻辑,是业务自己的事。

---

## 三层一句话

```
① Config / EyeToSpec   视觉事实真源:绝对 px + 左上角 + 对齐意图标注。静态、拍平、无逻辑。
        │  pack.json
        ▼
② 转换库(Config→Struct) 两件事:(a) load config → 转成跟 Phaser 对齐的布局 structure;
        │                      (b) 一批通用、无业务的布局函数(按需生长)。不认业务、不做决策、不渲染。
        ▼
③ 业务实现层            读 struct + 通用函数,做一切决策与逻辑:认重复区、屏幕缩放、
                        自动对齐、安全区消费、动态铺列表、状态、动效、数据绑定,最终上屏。
```

---

## ① Config / EyeToSpec —— 视觉事实

### 输出什么

一份忠于视觉稿的坐标清单(`pack.json`):
- `canvas`:整块画布 px(长条页就是一条长 strip,如 720×2200)。
- `background`:纯坐标 `{tex, x, y, w, h}`,和元素一样按 px 摆(`fit` 模式串是老 pack 兜底,新 pack 不用)。
- `elements`:每个元素的 **左上角 x/y + w/h + type + depth + detail(+ rotation/flipH/flipV)**,全是画布绝对 px。
- **对齐意图标注**:某元素是向上 HUD 对齐 / 底对齐 / 中心对齐 / 基线对齐。这是**视觉事实**(设计稿上就是这么定的),不是行为。
- `env`:设备外壳(frame 首屏视口窗、安全区、微信胶囊)——给 EyeToSpec 画,也供业务读。

### 明确不做

- 不存"怎么排"(几列、往哪续、铺几个)——那是排列**逻辑**。
- 不存"怎么动"(滚动、动效、状态变体)——视觉稿上没有时间维度。
- 不存元素之间的"关系"(父子、重复模板、内容撑高)——关系由业务反推。
- 不做任何坐标变换(不换中心点、不缩放、不锚定)。

### 为什么坐标是"左上角 px",不是中心 / 不是分数

- 分数要乘基准,不同元素挑了不同基准(屏宽/屏高/画布高)会**静默漂移**;px 只有一个基准=画布这块板。
- 左上角贴合出图直觉与 CSS/iOS/Android/SVG/设计工具 inspector;中心点是运行时(旋转缩放)才需要的,由 ② 换算。

---

## ② 转换库(Config → Struct)—— 只有两件事

### (a) load config → Phaser 布局 structure

把裸 JSON 转成业务顺手、跟 Phaser 概念对齐的结构:
- **坐标换成中心点 cx/cy**(`cx = x + w/2`)——业务从 ② 拿到的就是 Phaser 习惯的中心点,不自己碰左上角→中心。
- 一并带出 w/h、贴图文件路径、rotation/flipH、以及 ① 的**对齐意图标注**,让业务直接能喂 Phaser。
- 这个 structure 是**通用的、与具体页面业务无关**的。

### (b) 一批通用、无业务的布局函数

按需生长——**③ 真实使用中缺什么,就回头往 ② 补什么**。已知会用到的形态举例:
- 传几个元素 key 进来,算它们之间的间距 / 上下高度(供业务反推重复区规律)。
- 一步把安全区画出来这类现成便捷函数。
- px↔中心、等比缩放系数之类的纯几何工具。

### 明确不做

- **不认重复区**(哪些是一组样板、谁是相邻列)——③ 自己传 key 来问间距。
- **不做屏幕缩放决策**(缩不缩、缩多少)——③ 定。
- **不做锚定决策**(贴顶还是吸底)——③ 用 ① 的对齐标注自己算。
- **不消费 env**(安全区/frame 怎么用是业务的);② 顶多提供画安全区的通用函数。
- **不渲染、不碰业务数据。**

> 一句话:② 提供"能力",③ 做"决定"。② 把坐标和对齐标注梳理成好用的结构、把工具备好;要不要用、怎么组合、最终摆哪,全是 ③。

---

## ③ 业务实现层 —— 决策 + 逻辑 + 渲染

拿 ② 的 struct 和函数,把"视觉布局图"变换成"本页面 layout 里的真实逻辑"。这一步含业务逻辑,只是其中**布局那部分**靠 ② 转出的结构来喂。

职责:
- **认出重复区**:传相关 key 给 ②,算出一排的间距 / 间隔,再按真实数据(有几个塔)自己铺列表。
- **屏幕缩放**:③ 搞定。因 ① 已标注了每个元素向上 HUD / 底对齐 / 基线对齐,③ 据此**自动对齐**。
- **安全区 / frame 消费**:业务自己定怎么用(避让、留白)。
- **弹窗内动态跟随**(如属性面板跟着某个塔走)、状态变体、动效、数据绑定,全在这层。

---

## 两种消费方式共用同一份 Config(核心洞察)

Config 对"固定内容"和"动态内容"存的东西**完全一样**(都是具体实例),差别全在**业务怎么用**:

| 场景 | Config 存 | 业务怎么用 |
|---|---|---|
| **固定**(home 4 个 tab) | 4 个 tab 的具体位置 | 实例即真相,直接摆在它们说的位置 |
| **动态**(loadout 候选塔 1/2/3) | 首排那几张卡的具体位置 | 卡是**样板/相对位**,业务读它们**反推**出格宽/列距/行距,再按真实数据(N 个塔)自己铺 |

动态场景里,炮塔一/二/三**不是真实数据**,是用来算间距和间隔的样板。业务算出"一排的间距、一排的间隔"后,比着这个规律绘制真实内容列表。

---

## 弹窗定位定案

弹窗**不需要任何"父子跟随"机制**。分三种,各归各位:

1. **整体位置** = 拖到画布上的那个位置。框和子元素各自的绝对 px 已完全标明显示位置。
2. **屏幕缩放** = 整块弹窗(框 + 所有子元素)按一个对齐锚点吸附:top / bottom / **默认中心对齐**。框和子元素在画布系里相对位置本就固定,整块平移缩放天然一起动,不散架。
3. **框内动态跟随**(如塔的基础属性跟着那个塔走)= 纯业务逻辑,③ 自己写,Config 不表达。

**没有一种需要往 Config 加 parent 字段**——和排布规律同一个结论:关系不进 Config,由 ③ 处理。

---

## 缺能力往哪补:一条判据

以后发现少了某个能力,先问它是哪一类:

| 它是什么 | 补在哪 |
|---|---|
| **视觉事实**(画了什么、在哪、多大、谁压谁、对齐意图) | ① Config |
| **通用、无业务、确定性的布局工具**(算间距、换中心、画安全区) | ② 转换库 |
| **页面决策 / 逻辑**(怎么排、缩不缩、铺几个、怎么动、什么状态) | ③ 业务层 |

判"关系/规律"时的口诀:**Config 只存"画出来的实例",绝不存"规律"或"结果"。规律由 ③ 从实例反推,结果由 ③ 按数据算。**

---

## 当前结论:表达力够不够

**够。** 对"对齐视觉稿"这个目标,当前契约(canvas + 绝对 px 元素 + depth + 对齐标注 + env.frame)基本完备——因为视觉稿本就是拍平、静态、一屏一版的。

之前疑似的"缺口"(重复结构、弹窗父子、滚动、内容撑高)全都不是 Config 缺字段,而是被总原则解掉了:它们是**关系/规律/行为**,归 ② 或 ③。

**真正还没做的,不是 Config 的表达力,而是:**
- **② 转换库尚不存在**——契约里的 px→中心、屏幕缩放辅助、通用函数,全是"声明了、待实现"。这是把今天所有 Config 成果接进游戏的唯一桥。
- **③ 还没有一处真正消费新 pack.json**。游戏侧现在读的仍是旧的归一化 `*-layout.json` + 各写各的 resolver。需要选一页(loadout 最合适)走通端到端:读绝对 pack → ② 转换 → Phaser 上屏。
- **battle 棋盘尚未 config 化**(核心几何还硬编码在 `board-renderer.ts` 的 `_FACTOR` 常量)——这是工作量,不是表达力问题。

---

## 业务层义务(原则)+ 落地机制

边界宪法保证业务**能做什么**(灵活),这一节保证业务**不能乱做什么**——把自由锁在"数据 + 逻辑 + 交互"上,不让它漏到"布局真源 / 坐标变换"上。目的是**杜绝业务各造各的轮子**(battle 的 18 条 `_FACTOR` 硬编码就是反面教材)。

### 五条原则

- **P1. Config 是布局真源的唯一出口。** 业务用到的任何布局数值,必须能追溯回某个 Config 元素;代码里不准出现凭空的坐标/尺寸常量。缺了就补进 Config,不在代码里写死。
- **P2. 单一解析路径:只有 ② 读 pack,业务永不直接解析。** 业务只跟 ② 的 struct 打交道,杜绝各页各种读法。
- **P3. 变换只有一份实现,在 ②;业务不重造。** 坐标数学(换中心、缩放、算间距)发现自己在写,就是信号——调 ② 或给 ② 补,而不是自己算。
- **P4. 逃生舱只有一个方向:缺能力 → 补进对应层,永不在代码里凑。** 缺视觉信息补 ①,缺通用工具补 ②。一旦允许"先在代码里凑一下",真源就开始烂。
- **P5. 跨层靠约定命名对接,命名即接口。** ② 靠命名认结构、③ 靠命名认抓手,两边守同一套约定。

一句话精神:**业务的自由 = 数据 + 逻辑 + 交互;业务的不自由 = 布局真源、坐标变换。凡是"布局"性质的,一律回 ①/②,业务不自建。**

### 落地机制:优先级 机器 > 结构 > 人

- **主力 = 架构隔离(锁 P2/P3):业务物理上不能直接读 `pack.json`。** ② 是唯一能碰 pack 数据的包,业务层引不到 pack 路径 / 解析器 / fs 读 json——把"要求"变成"物理不可能违反"。可配 ESLint `no-restricted-imports` 加固。这是唯一"不靠人自觉、改一次结构永久生效"的手段,是核心。
- **数据入口校验(锁 P5):** ② load config 时顺手校验跨层命名约定,不合规直接报错,在最早处暴露。成本极低。
- **Review 清单兜底(P1/P4):** 机器管不了的才落到人。
- **落地时机:随首个端到端样板(loadout)走通时立规矩**——建 ② 时就把隔离结构做进去,不提前写 lint 空转(没有真实样本容易误报),晚了业务乱引一通再收更疼。

## 尚未讨论完的话题(留待后续)

- **弹性区**:一段空间在上下两条 `line` 之间,按屏幕高度对若干 `group` 整组伸缩(loadout `runtime.elasticZone` 是现场)。已明确"line 的 id、group 的 name 是跨层接口锚点,③ 靠命名认出抓手、注入适配行为",但具体约定待细化。
- **滚动语义**:canvas 比 frame 高即隐式下滚。是否需要显式字段,还是纯归 ③,待定。
- **`runtime` / `label` 的定位**:这些"半运行时语义"字段按总原则应降为**给业务的可选提示**,不由 Config 承担行为语义;边界待正式清理。
