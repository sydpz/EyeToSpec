# EyeToSpec 架构定稿 — 绝对坐标契约(2026-07-13)

一天之内把 EyeToSpec 从「归一化 + 模式字符串」的旧世界,收敛到一套**单一真源、零再派生**的绝对坐标契约。这份文档记录**为什么这么定**,而不只是定了什么。契约细节见 [`absolute-contract.md`](absolute-contract.md);本文是它背后的思路与今天这条演进线。

## 一句话定位

EyeToSpec 是**纯静态叠图器**:读 `canvas + elements + env`,按 `depth` 从低到高画,画完就完。**任何"适配真机"的数学都不在这里**——归一化、中心点换算、屏幕缩放、滚动/锚定,全是下游运行时库的活。这条分界线是今天所有决策的总纲。

## 三层架构(职责铁律)

```
① Config(EyeToSpec 产物)   ——  绝对坐标契约:px + 左上角,一切皆 element
        │  pack.json:canvas / background / elements / env / runtime
        ▼
② 转换库(公共运行时,待建)  ——  归一化到真机 · 左上角→中心 · fit/anchor/scroll · 按 group 缩放
        │
        ▼
③ 业务/游戏层               ——  喂给 Phaser,不碰坐标语义
```

- **① 只存坐标,不懂业务**。一根 `line` 是什么意思(滚动区顶?背景对齐线?)Config 不知道,由 ② 按元素 id 解释。
- **② 是唯一做变换的地方**,躲在单测后面。
- 每一块 top-level 字段各有受众,互不踩:`background`+`elements` 是共享真源;`env` 只给 EyeToSpec;`runtime` 只给游戏。各自跳过不属于自己的块。

## 今天的演进线(按提交顺序,含思路)

### 1. px + 左上角全链贯通(`40bd4c4`)

**问题**:旧契约存 `0.xxx` 分数 + 中心原点。分数必须乘某个基准,而不同元素挑了不同基准(屏宽 / 屏高 / 画布高),数值**静默漂移**,对不上时根本分不清错在值还是错在换算。

**决策**:坐标一律**绝对像素 + 左上角原点**。
- 像素只有一个基准——画布这块板,能一眼读懂、能有意义地 diff 两份文件。
- 左上角贴合出图直觉("素材左上角落这")、贴合 CSS/iOS/Android/SVG/设计工具 inspector。EyeToSpec 是**创作工具**,就说创作端的话。
- 中心点是**运行时**的事(旋转/缩放好用),由 ② 做 `centerX = x + w/2`,文件永远存左上角。

同批:element 加 `label` 分层标注(哪层:overlay / scroll),`depth` 可编辑。

### 2. Save 写回真 pack + 分组 + 单元素居中 + 文字写回(`ce7098c`)

EyeToSpec 从"只读预览"变成"能改能存"。
- **分组 `group`**(扁平不嵌套,一元素至多一组):选中任一成员即选整组,对齐时**整组当刚体**按合并包围盒平移,组内相对位置不变。运行时用它做**按组合结构缩放**(而非逐 item 缩)——所以组归属必须持久化进元素数据。
- 对齐/分布改为按**单元**(每个 group 一个单元,未分组元素各自一个单元),纯未分组多选行为退化回原逻辑不变。

### 3. Phone-frame fit — env.frame 双字段 + 侧边常驻面板(`6f12bfc`→`9cb4412`)

**问题**:长条滚动页里,"手机屏幕视口"到底框住哪一段?旧做法把 dialog 直接怼着面板画,丢了"占屏多大 / 在哪"。

**决策**:`env.frame` 是可以**自由摆在板上任意位置**的手机视口矩形(不必贴顶)——长条页里它就是首屏那一刀,底下是滚进来的内容。
- `env` 是**声明式设备外壳**:配了就画,没配就不画;一切挂在 `frame` 上,没 `frame` 则 `safeTop/safeBottom/wxCapsule` 全不画(没屏幕可挂)。
- `wxCapsule` 按 `frame.w / basisW` **只随宽缩**——微信胶囊是物理固定尺寸,用高度分数会在 1280 与 1600 高的屏之间漂移。
- 写回按 `frameDirty` 门控,保留 `wxCapsule/name`。

### 4. 基准线收敛为普通 `line` 元素(`3088191`→`90bbb8a`)

**问题**:旧有 `anchorLine` / `elasticZone` 这类专门机制,是特例、是历史包袱。

**决策**:删掉专门机制,基准线就是一个 `type:"line"` 的**普通元素**(px 坐标,结构上=一根细横 box:`x=0`、`w=画布宽`、`h=线厚`,只有 `y` 有语义)。**key 即标识**(如 `anchor-line`)。
- Config 只存位置,**这根线_意味着什么_由下游按 id 解释**——契约不背业务语义。这正是"① 只存坐标"铁律的样板落地。

### 5. background 纯坐标形式(`24a36b6`)—— 今天的收口

**问题(今天真踩的坑)**:loadout 长条羊皮纸背景死活不显示。查下来:单张 `background` 少了 `fit` 字段就走 `contain` 分支,把 4096 高的图缩进画布 → 看不到长条。加了 `fit:width-top` 还是空——**真因根本不是 fit**,是图文件不在 pack 的 `repo` 路径下(404)。

**由此看清**:`fit` / `topCy` / `repeat` / `cover` 这套**模式字符串本身就是历史逻辑**——背景在靠字符串"猜"怎么铺,和"一切皆 px 元素"的契约不一致。

**决策**:背景也走**纯坐标**,跟 element 一模一样。
```json
"background": { "tex": "loadout-scroll-bg", "x": 0, "y": 0, "w": 720, "h": 2200 }
```
给 x/y/w/h,就按这个框铺上去(`backgroundSize:100% 100%`),用画布百分比定位所以缩放/resize 自动跟随。超长背景 = `h > canvas` 往下溢出,滚动画布天然正确,**无需任何 fit 语义**。`fit`(width-top/width-bottom/layers)保留成老 pack 的兜底分支,不动存量。

> 附带减弱了后续开发复杂度:头图不再是"背景第二层",而是一个高 `depth` 的普通 image 元素叠在长条背景上——又一次"一切皆 element"。

## loadout 页最终形态(样板)

- 旧文档有两份(base 顶弹性 + panel 浮层)。先各自独立转成新契约 pack 校准,再**手工拼**成一份最完整的长条 `loadout` pack(720×2200)。
- 拼接完成后,两个中间产物 pack + 生成脚本删除;素材(`loadout-scroll-bg.jpg`)入库主 repo 提交 main。
- 背景 = 单张长条羊皮纸(纯坐标);头图(`hutHead`)= 高 depth 的 image 元素叠在上面。

## 不变量清单(所有决策的共同底座)

1. **坐标一律 px + 左上角**,文件里不出现 `0.xxx` 分数。
2. **一切皆 element**(线、头图、背景都往这个模型靠)。
3. **EyeToSpec 不做适配数学**,只静态叠图;归一化/中心换算/fit/anchor/scroll/按组缩放全在 ②。
4. **Config 不背业务语义**;line/label/group 的"含义"由下游按 id/tag 解释。
5. **单一真源,零再派生**:同一份文件,编辑器画的和运行时读的完全一致。

## 遗留与后续

- `fit` / `layers` / `topCy` 分支仍为存量 pack 兜底;新 pack 一律纯坐标 background。
- **转换库(②)尚未建**——归一化、中心换算、fit/anchor/scroll、按 group 缩放都还是"契约里声明、下游待实现"。
- 存量老 pack 的迁移(分数→px)靠生成器一次性烤死,别手算。
